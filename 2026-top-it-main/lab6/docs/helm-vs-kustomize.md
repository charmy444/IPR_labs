# Helm и Kustomize: сравнение и выбор подхода

Оба инструмента решают одну общую задачу — **параметризовать и переиспользовать** Kubernetes-манифесты для разных окружений и версий. Различаются **моделью данных**, **точкой расширения** и **экосистемой**.

Кратко:

- **Kustomize** оставляет исходники в виде «обычного» API Kubernetes и **накладывает слои** (патчи, генераторы, замены).
- **Helm** упаковывает манифесты в **шаблоны** и отделяет **данные** (values) от **логики разметки** (templates).

Ниже — различия, компромиссы и практические сценарии.

---

## Сравнение «в одной таблице»

| Критерий | Kustomize | Helm |
|----------|-----------|------|
| Исходные манифесты | Валидные YAML, как для `kubectl apply` | Шаблоны с `{{ }}`, итог — YAML после рендера |
| Параметризация | Патчи, overlays, `images`, генераторы | `values.yaml`, `--set`, вложенные структуры |
| Окружения | Каталоги `overlays/dev`, `overlays/prod` | Файлы `values-dev.yaml` / `values-prod` или отдельные релизы |
| Единица установки | Нет встроенного «релиза»; вы применяете результат `kustomize build` | **Release** с именем, историей, откатом |
| Зависимости | Нет аналога subchart; композиция через `resources` и внешние base | `dependencies` в `Chart.yaml`, umbrella charts |
| Ревью в Git | Проще отследить дифф «какой YAML изменился» в base/patch | Нужно читать шаблон + values; дифф итоговых манифестов — через `helm template` |
| Порог входа | Ниже для тех, кто уже пишет голые манифесты | Выше: шаблоны Go, хелперы, конвенции чартов |
| Типичная интеграция | GitOps (Flux нативно, Argo CD с Kustomize) | Helm-релизы (Argo CD Helm, Flux HelmRelease) |

## Различие подходов (концептуально)

### Kustomize: композиция и патчи

Вы мыслите **наследованием от базы**: «в prod мы к базе добавляем только то, чем prod отличается». Логика условий (`if prod then …`) в манифестах **не выражается языком шаблонов** — её имитируют **отдельными файлами** и **отдельными overlays**. Это снижает риск «запутанного шаблона», но при большом числе исключений дерево overlays может разрастись.

### Helm: пакет и шаблон

Вы мыслите **пакетом с параметрами**: один chart, много комбинаций values. Условия и циклы **можно** выразить в шаблоне (`if`, `range`), что экономит файлы, но ухудшает читаемость при злоупотреблении. Сильная сторона — **версионирование чарта**, **откат релиза**, **зависимости**, богатая экосистема готовых чартов.

## Что и когда использовать

### Когда уместен Kustomize

- Команда хочет **минимум магии**: манифесты должны выглядеть как документация к API Kubernetes.
- Несколько окружений отличаются **умеренно** (ресурсы, реплики, лейблы, namespace).
- Внедрён **GitOps**, где удобно хранить **плоские** и **патченные** YAML (Flux «Kustomization», Argo с kustomize).
- Нужно **быстро** разделить общий base между командами (общий каталог base, свои overlays).

### Когда уместен Helm

- Нужен **единый пакет** для внутренних или внешних потребителей (установка «в одну команду» с values).
- Важны **история и откат** релиза на уровне пакета.
- Есть **зависимости** (например несколько сервисов + общие subcharts) — осторожно с границами ответственности (см. лаб. 6: БД не обязательно в том же чарте, что приложение).
- Организация уже стандартизировала **Helm** для поставки в кластеры.

### Когда рассматривают оба

- Разные слои: например **Helm** для стороннего ПО из репозитория чартов, **Kustomize** для «нашего» слоя патчей поверх сгенерированного или vendored YAML.
- **Helm post-renderer** с Kustomize: chart рендерится, затем к результату применяется kustomize (усложняет отладку, но встречается в энтерпрайзе).

## Типичные ошибки выбора

1. **Затащить всю инфраструктуру в один app-chart** «ради удобства», включая БД — нарушение границ жизненного цикла и секретов (в курсе это сознательно разведено).
2. **Раздувать Helm-шаблон** сотнями `if` вместо второго values-файла или subchart — лучше упростить модель.
3. **Дублировать** одно и то же **и** огромным Helm chart, **и** десятком Kustomize-патчей без договорённости — выберите **один** основной источник правды для команды.
