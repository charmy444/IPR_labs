# Scope Sprint 1: QA and Acceptance Boundaries

## 1. Цель
Зафиксировать границы проверок, критерии приемки и обязательные негативные сценарии для Sprint 1, чтобы команда одинаково понимала, что считается завершенным по качеству.

## 2. Область покрытия
1. Генерация CSV по преднастроенным шаблонам.
2. Анонимизация любого загруженного CSV по правилам колонок.
3. Скачивание результата в сценариях Generate/Anonymize.
4. Базовая обработка ошибок для невалидного ввода.

## 3. Вне области спринта 1
1. Нагрузочные и перфоманс-тесты.
2. Полноценное security/pentest-тестирование.
3. Проверка качества Similar/Synthesize (этап после MVP).

## 4. Уровни тестирования
1. Контракт/API: HTTP-статусы, форматы запросов и ответов.
2. Функциональные проверки: структура CSV, число строк, применение правил.
3. Негативные проверки: ошибочные параметры, методы, файлы.
4. Smoke/E2E: базовые сквозные сценарии.

## 5. Критерии входа и выхода
### Критерии входа
1. Согласованы документы: `scope`, `architecture`, `api-contracts`, `schemas`, `templates`.
2. Эндпоинты доступны в локальной среде.

### Критерии выхода
1. Все критерии приемки из раздела 6 выполнены.
2. Нет blocker/critical-дефектов в ветках MVP.
3. Чеклист smoke из раздела 8 выполняется без устной передачи знаний.

## 6. Критерии приемки

### 6.1 Генерация CSV
1. При валидном запросе `POST /api/v1/generate/run` API возвращает `200`.
2. Ответ приходит в JSON и содержит `result_format`.
3. Для одного шаблона backend возвращает `csv_base64` в `content_base64`.
4. Для нескольких шаблонов backend возвращает `zip_base64` в `archive_base64`.
5. Заголовок сгенерированного CSV строго соответствует схеме выбранного шаблона.
6. Количество строк данных точно равно `row_count`.
7. Неизвестный `template_id` или нарушенные зависимости -> документированный 4xx.
8. Невалидный `row_count` -> `422`.

### 6.2 Анонимизация любого CSV
1. При валидном CSV и валидных `rules` API возвращает `200`.
2. Порядок колонок в выходном CSV совпадает с входным.
3. Количество строк в выходном CSV равно входному.
4. В MVP поддерживаются методы `keep`, `mask`, `redact`, `pseudonymize`, `generalize_year`.
5. Метод `keep` не меняет значение.
6. Метод `mask` изменяет исходное значение на маскированный вид.
7. Метод `redact` скрывает значение по контракту.
8. Метод `pseudonymize` работает консистентно: одинаковый вход -> одинаковый выход в пределах колонки.
9. Неизвестная колонка в `rules` -> `400/422` с указанием колонки.
10. Неподдерживаемый `method` -> `400/422` с подсказкой по допустимым методам.

### 6.3 Сценарий скачивания результата
1. После успешного `POST /api/v1/generate/run` пользователь может скачать CSV или ZIP, собранный из `base64`.
2. После успешного `POST /api/v1/anonymize/run` пользователь может скачать CSV.
3. Скачанный файл открывается как валидный CSV.
4. Имя файла в JSON-ответе непустое и содержит корректное расширение.
5. Файл объемом от 100 строк скачивается без обрезания.

## 7. Негативные кейсы (обязательные)

### 7.1 CSV и входные данные
1. Пустой файл -> `400/422`, явная ошибка «пустой CSV».
2. Только заголовок без строк данных -> стабильное задокументированное поведение.
3. Поврежденный CSV (кавычки/переносы) -> `400/422`.
4. Плохой delimiter (например, `;` вместо `,`) -> детерминированное поведение:
- при автоопределении разделителя разбор проходит;
- без автоопределения возвращается ошибка разбора.
5. Не-CSV файл с расширением `.csv` -> `400/422` при ошибке парсинга.

### 7.2 API
1. Неправильный HTTP-метод -> `405 Method Not Allowed`.
2. Отсутствуют обязательные поля -> `422 Unprocessable Entity`.
3. Неверные типы полей (`row_count` строкой, `rules` невалидной структурой) -> `422`.
4. Слишком большой запрос (если настроен лимит) -> `413` или документированный эквивалент.

### 7.3 Доменная логика
1. Неизвестный шаблон в Generate -> `400/422`.
2. Неверный метод анонимизации (`md5`, `encrypt` и т.п.) -> `400/422`.
3. Конфликтующие настройки по одной колонке -> предсказуемая ошибка валидации.

## 8. Чеклист smoke для спринта 2
1. `GET /health` возвращает `200`.
2. Generate: запрос -> скачивание -> проверка заголовка и числа строк.
3. Anonymize: загрузка CSV + правила -> скачивание -> проверка числа строк и трансформаций.
4. Негативный smoke: неправильный HTTP-метод -> `405`.
5. Негативный smoke: пустой CSV -> ожидаемая ошибка клиента.

## 9. Базовые E2E-сценарии
1. Generate `users` на 100 строк и проверка схемы/количества.
2. Загрузка `users.csv` в Anonymize, смешанные правила (`keep/mask/redact/pseudonymize/generalize_year`), проверка результата.
3. Загрузка поврежденного CSV и проверка не-200 ответа с понятной ошибкой.
