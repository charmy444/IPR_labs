# Модель данных и связи (MVP)

## 1. Назначение
Документ определяет логику взаимодействия между встроенными шаблонами генерации CSV и устанавливает обязательный порядок их создания для обеспечения ссылочной целостности данных (Foreign Keys: `user_id`, `order_id`, `product_id`).

В MVP поддерживаются 5 шаблонов:
* **users** — базовая таблица.
* **products** — независимая таблица.
* **orders** — зависит от `users` и `products`.
* **payments** — зависит от `orders` и `users`.
* **support_tickets** — зависит от `users`.

## 2. Связи между таблицами

### `users`
Центральная таблица субъектов.
* **Первичный ключ**: `user_id`.

### `orders`
Связывает транзакции с пользователями.
* **Связь**: `orders.user_id -> users.user_id`.
* **Связь**: `orders.product_id -> products.product_id`.
* **Требование**: Каждый заказ должен быть привязан к существующему пользователю и существующему товару.

### `payments`
Фиксирует оплату заказов.
* **Связи**:
    * `payments.order_id -> orders.order_id`
    * `payments.user_id -> users.user_id`
* **Логика согласованности**: Поле `payments.user_id` должно строго соответствовать идентификатору пользователя, которому принадлежит выбранный заказ `payments.order_id`.

### `support_tickets`
Журнал обращений в поддержку.
* **Связь**: `support_tickets.user_id -> users.user_id`.

### `products`
Каталог товаров.
* **Связи**: В рамках MVP независим от других сущностей.

## 3. Порядок генерации
Для исключения «висячих» ссылок генерация должна выполняться в следующей последовательности:

1. **users** и **products** (независимые сущности).
2. **orders** (требует наличия `users` и `products`).
3. **payments** (требует наличия `orders`).
4. **support_tickets** (требует наличия `users`).

## 4. Контекст генерации
В процессе работы backend-модуль сохраняет идентификаторы созданных записей во внутреннем объекте контекста.

**Пример структуры контекста:**
```json
{
    "users": [
        {"user_id": 1},
        {"user_id": 2}
    ],
    "orders": [
        {"order_id": 1, "user_id": 1},
        {"order_id": 2, "user_id": 2}
    ]
}
```

## 5. Правила работы со связями
* **Для `orders`**: Значение `user_id` выбирается случайным образом из накопленного списка в контексте пользователей.
* **Для `orders`**: Значение `product_id` выбирается случайным образом из накопленного списка в контексте товаров.
* **Для `payments`**: Из контекста выбирается случайный объект заказа. Значения `payments.order_id` и `payments.user_id` копируются из выбранного объекта одновременно для обеспечения консистентности.
* **Для `support_tickets`**: Значение `user_id` выбирается случайно из контекста пользователей.

## 6. Обработка ошибок целостности
* **Запрет фиктивных данных**: Использование заглушек (например, "N/A") в полях внешних ключей недопустимо.
* **Отсутствие контекста**: Если шаблон требует внешний ключ, а соответствующий контекст пуст, генерация прерывается с вызовом `GenerationError`.
* **Нарушение порядка**: Попытка генерации зависимых таблиц (например, `payments`) без предварительного создания родительских (`orders`) считается ошибкой сценария.

## 7. Режимы работы MVP
1. **Пакетная генерация**: Backend автоматически выстраивает цепочку создания таблиц согласно правилам зависимости.
2. **Одиночная генерация**: Требует наличия предзаполненного контекста, иначе возвращает ошибку.

## 8. Критерии корректности (Definition of Done)
Результат генерации считается валидным, если:
* Все `orders.user_id` присутствуют в таблице `users`.
* Все `orders.product_id` присутствуют в таблице `products`.
* Все `payments.order_id` присутствуют в таблице `orders`.
* Значение `payments.user_id` в каждой строке соответствует владельцу конкретного заказа.
* Все `support_tickets.user_id` существуют в системе.
