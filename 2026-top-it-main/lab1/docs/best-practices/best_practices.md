# Лучшие практики работы с Git и GitLab

## Коммиты

### Частые и атомарные коммиты

Делайте коммиты часто и маленькими порциями. Каждый коммит должен представлять одну логическую единицу работы.

```bash
# Хорошо: атомарные коммиты
git add login-form.js
git commit -m "Добавлена форма входа"

git add login-validation.js
git commit -m "Добавлена валидация формы входа"
```

### Осмысленные сообщения коммитов

Пишите информативные сообщения коммитов, которые отвечают на вопросы "что?" и "почему?":

#### Структура сообщения коммита

```
Краткое описание (до 50 символов)

Более подробное описание, если необходимо. Объясните, что было
сделано и почему. Можно описать проблемы, которые решает этот
коммит, и как он влияет на проект.

Если есть связанные задачи, укажите их:
Closes #123
Refs #456
```

#### Примеры хороших сообщений

```
Добавлена валидация email в форме регистрации

Реализована проверка корректности email-адреса перед отправкой
на сервер. Это предотвращает ошибки при отправке уведомлений
зарегистрированным пользователям.

Closes #789
```

```
Исправлено падение приложения при пустом массиве данных

Исправлена ошибка в функции renderList, которая вызывала
исключение при попытке отрисовать пустой список элементов.

Fixes #101
```

### Использование соглашений по именованию

Используйте глаголы в повелительном наклонении:

- ✅ "Добавить функцию поиска"
- ✅ "Исправить ошибку валидации"
- ✅ "Обновить зависимости"
- ❌ "Добавлена функция поиска"
- ❌ "Исправлена ошибка"

## Ветвление

### Стратегия именования веток

Следуйте согласованной стратегии именования веток:

```
feature/описание-функции
bugfix/описание-ошибки
hotfix/описание-срочного-исправления
refactor/описание-рефакторинга
docs/описание-документации
```

### Держите ветки короткоживущими

Чем дольше ветка существует, тем больше вероятность конфликтов при слиянии. Стремитесь к тому, чтобы feature-ветки жили не более нескольких дней.

### Частая синхронизация с основной веткой

Регулярно обновляйте свою ветку относительно main:

```bash
git fetch origin
git rebase origin/main
```

## Merge Requests

### Создавайте маленькие MR

Один Merge Request должен решать одну задачу. Это упрощает код-ревью и снижает вероятность ошибок.

### Пишите качественные описания

Хорошее описание MR включает:

1. **Что было сделано**
2. **Зачем это нужно**
3. **Как проверить изменения**
4. **Скриншоты (если применимо)**

### Отвечайте на комментарии

Не игнорируйте комментарии ревьюеров. Даже если вы не согласны, объясните свою точку зрения вежливо и конструктивно.

## Работа с историей

### Используйте интерактивное перебазирование

Для очистки истории перед слиянием используйте интерактивное перебазирование:

```bash
git rebase -i HEAD~3
```

Это позволяет:
- Объединить коммиты (squash)
- Переписать сообщения коммитов
- Удалить ненужные коммиты
- Переупорядочить коммиты

### Избегайте force push в общие ветки

Никогда не используйте `git push --force` в общие ветки (`main`, `develop`), так как это может повредить историю для других разработчиков.

## Конфигурация Git

### Настройте алиасы

Создайте алиасы для часто используемых команд:

```bash
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'
git config --global alias.visual '!gitk'
```

### Используйте .gitignore

Создайте и поддерживайте актуальным файл `.gitignore`:

```
# Зависимости
node_modules/
vendor/

# Сборка
dist/
build/
*.min.js
*.min.css

# Среда разработки
.env
.vscode/
.idea/

# ОС
.DS_Store
Thumbs.db
```

## Безопасность

### Не храните чувствительные данные

Никогда не коммитьте:
- Пароли
- Ключи API
- Токены доступа
- Приватные ключи

Используйте переменные окружения или специальные инструменты для управления секретами.

### Регулярно обновляйте SSH-ключи

Устанавливайте срок действия SSH-ключей и обновляйте их регулярно.

## Работа в команде

### Коммуникация

- Согласовывайте работу над пересекающейся функциональностью
- Используйте issue tracker для обсуждения задач
- Документируйте важные архитектурные решения

### Code Review

- Проводите код-ревью всех изменений
- Будьте конструктивны в комментариях
- Учитесь у коллег во время ревью

### Документация

- Поддерживайте документацию в актуальном состоянии
- Документируйте неочевидные решения
- Пишите комментарии в коде, где это необходимо

## Инструменты и автоматизация

### Используйте Git hooks

Настройте клиентские и серверные хуки для автоматизации проверок:

- pre-commit: проверка кодстайла, запуск тестов
- pre-push: финальная проверка перед отправкой

### CI/CD

Интегрируйте непрерывную интеграцию для автоматического запуска:
- Тестов
- Проверок кодстайла
- Статического анализа
- Безопасностных проверок

## Работа с ветками

### Создание и переключение между ветками

```bash
# Создание новой ветки и переключение на нее
git checkout -b feature/new-functionality

# Переключение на существующую ветку
git checkout main

# Просмотр всех веток
git branch -a
```

### Работа с ветками в команде

1. Всегда создавайте ветки от актуальной версии main
2. Называйте ветки в соответствии с соглашением: feature/описание, bugfix/описание
3. Делайте частые коммиты и отправляйте изменения на сервер
4. Перед созданием Merge Request обновите ветку относительно main
