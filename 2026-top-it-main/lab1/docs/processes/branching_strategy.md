# Стратегия ветвления

## Введение

Стратегия ветвления — это набор правил и соглашений о том, как и когда создавать ветки, как их именовать и как объединять с основной веткой. Хорошая стратегия ветвления помогает поддерживать чистую историю проекта и упрощает совместную разработку.

## Основные типы веток

### Main/Branch (основная ветка)

- **Назначение**: Содержит стабильную версию кода, готовую к production
- **Имя**: `main` или `master`
- **Обновление**: Только через Merge Request после тщательного код-ревью
- **Статус**: Всегда должна быть рабочей

### Feature Branches (ветки функциональности)

- **Назначение**: Разработка новых функций или улучшений
- **Имя**: `feature/описание-функции`
- **Примеры**: `feature/user-authentication`, `feature/payment-integration`
- **Жизненный цикл**: Создается от `main`, после завершения сливается обратно в `main`

### Bugfix Branches (ветки исправления ошибок)

- **Назначение**: Исправление ошибок в основном коде
- **Имя**: `bugfix/описание-ошибки`
- **Примеры**: `bugfix/login-error`, `bugfix/calculation-fix`
- **Жизненный цикл**: Создается от `main`, после исправления сливается обратно в `main`

### Hotfix Branches (ветки срочных исправлений)

- **Назначение**: Срочные исправления в production-коде
- **Имя**: `hotfix/описание-срочного-исправления`
- **Примеры**: `hotfix/security-patch`, `hotfix/critical-bug`
- **Жизненный цикл**: Создается от `main`, сливается в `main` и в текущую feature-ветку

### Release Branches (ветки релизов)

- **Назначение**: Подготовка к выпуску новой версии
- **Имя**: `release/v1.0.0`
- **Использование**: В учебных проектах обычно не требуется

## Соглашения по именованию веток

### Префиксы

Используйте префиксы для быстрой идентификации типа ветки:

```
feature/    # Новая функциональность
bugfix/     # Исправление ошибок
hotfix/     # Срочные исправления
experiment/ # Экспериментальные изменения
refactor/   # Рефакторинг кода
docs/       # Изменения в документации
```

### Стиль именования

1. **Используйте kebab-case** (через дефис):
   - ✅ `feature/user-profile-page`
   - ❌ `feature/userProfilePage` или `feature/user_profile_page`

2. **Будьте описательны, но кратки**:
   - ✅ `feature/add-password-reset`
   - ❌ `feature/fix` или `feature/changes`

3. **Избегайте специальных символов**:
   - ✅ `bugfix/login-validation-error`
   - ❌ `bugfix/login#validation@error`

## Процесс работы с ветками

### Создание ветки

1. Перед созданием ветки убедитесь, что у вас актуальная версия `main`:

```bash
git checkout main
git pull origin main
```

2. Создайте новую ветку от `main`:

```bash
git checkout -b feature/your-feature-name
```

### Работа в ветке

1. Делайте частые, атомарные коммиты:

```bash
git add file1.js
git commit -m "Добавлена валидация email"
```

2. Регулярно отправляйте изменения на сервер:

```bash
git push origin feature/your-feature-name
```

### Слияние ветки

1. Перед слиянием обновите свою ветку относительно `main`:

```bash
git checkout feature/your-feature-name
git rebase main
```

2. Отправьте обновленную ветку на сервер:

```bash
git push origin feature/your-feature-name --force
```

3. Создайте Merge Request через веб-интерфейс GitLab

## Управление ветками

### Просмотр веток

Просмотр локальных веток:

```bash
git branch
```

Просмотр всех веток (локальных и удаленных):

```bash
git branch -a
```

### Переключение между ветками

```bash
git checkout branch-name
```

### Удаление веток

Удаление локальной ветки:

```bash
git branch -d branch-name
```

Удаление удаленной ветки:

```bash
git push origin --delete branch-name
```

## Best Practices по ветвлению

### 1. Создавайте ветки для каждой задачи

Каждая логическая единица работы должна находиться в отдельной ветке.

### 2. Держите ветки короткоживущими

Чем дольше ветка существует, тем больше вероятность конфликтов при слиянии.

### 3. Часто синхронизируйтесь с main

Регулярно обновляйте свою ветку относительно `main`:

```bash
git fetch origin
git rebase origin/main
```

### 4. Используйте описательные имена

Имя ветки должно четко отражать её назначение.

### 5. Следуйте принятой в команде стратегии

Если в вашем проекте уже есть соглашения по ветвлению, следуйте им.
