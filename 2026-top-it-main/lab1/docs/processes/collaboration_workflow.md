# Рабочий процесс совместной разработки

## Введение

Совместная разработка с использованием Git и GitLab основана на распределенной модели, где каждый участник имеет свою локальную копию репозитория. Этот подход позволяет работать параллельно над различными частями проекта, а затем объединять изменения через систему Merge Request.

## Роли в команде

### Разработчик проекта (владелец репозитория)

Ответственности:
- Создание и настройка репозитория
- Добавление участников в проект
- Проверка и слияние Merge Request
- Поддержание структуры проекта и кодстайла

### Участник проекта (collaborator)

Ответственности:
- Клонирование репозитория
- Создание feature-веток для новых функций
- Реализация функциональности
- Создание Merge Request для код-ревью
- Участие в обсуждении изменений

## Этапы совместной разработки

### Этап 1: Подготовка

1. **Разработчик проекта** создает репозиторий на gitlab.mai.ru
2. **Разработчик проекта** добавляет участника(ов) с ролью Developer
3. **Участник проекта** клонирует репозиторий:

```bash
git clone git@gitlab.mai.ru:username/project-name.git
cd project-name
```

### Этап 2: Синхронизация

Перед началом работы всегда обновляйте локальную версию:

```bash
git checkout main
git pull origin main
```

### Этап 3: Создание feature-ветки

Создайте отдельную ветку для своей функциональности:

```bash
git checkout -b feature/description-of-feature
```

### Этап 4: Работа над функциональностью

1. Внесите изменения в код
2. Добавьте измененные файлы в индекс:

```bash
git add changed-file.js
```

3. Создайте коммит с описательным сообщением:

```bash
git commit -m "Добавлена валидация входных данных"
```

### Этап 5: Отправка изменений

Отправьте изменения на сервер:

```bash
git push origin feature/your-feature
```

### Этап 6: Создание Merge Request

1. Перейдите в веб-интерфейс GitLab
2. Создайте новый Merge Request:
   - Source branch: ваша feature-ветка
   - Target branch: main
   - Добавьте подробное описание изменений
   - Назначьте ревьюера (разработчика проекта)

### Этап 7: Код-ревью

**Разработчик проекта** проверяет предложенные изменения:
- Соответствие кодстайлу проекта
- Корректность реализации функциональности
- Отсутствие побочных эффектов
- Наличие комментариев в коде

При необходимости оставляются комментарии с предложениями по улучшению.

### Этап 8: Слияние изменений

После успешного код-ревью и разрешения всех комментариев:
1. **Разработчик проекта** выполняет слияние через веб-интерфейс GitLab
2. Рекомендуется выбирать опцию "Delete source branch" для поддержания чистоты репозитория

### Этап 9: Обновление локального репозитория

Оба участника обновляют свои локальные репозитории:

```bash
git checkout main
git pull origin main
```

## Управление участниками проекта

### Добавление участника

1. В веб-интерфейсе GitLab перейдите в Settings → Members
2. Введите username партнера
3. Выберите роль "Developer"
4. Установите срок действия доступа (рекомендуется 2-4 недели)
5. Участник получит email-уведомление о приглашении

### Изменение прав доступа

Роли в GitLab:
- **Guest** — минимальные права, просмотр
- **Reporter** — чтение, создание issues
- **Developer** — push в репозиторий, создание MR
- **Maintainer** — управление проектом, добавление участников
- **Owner** — полные права на группу/проект

## Синхронизация с удаленным репозиторием

### Получение изменений

Для получения последних изменений из удаленного репозитория:

```bash
git fetch origin
```

Для получения и слияния изменений:

```bash
git pull origin main
```

### Отправка изменений

Для отправки локальных изменений на сервер:

```bash
git push origin branch-name
```

## Работа с несколькими участниками

Когда над проектом работает несколько человек:
1. Всегда обновляйте локальную копию перед началом работы
2. Создавайте отдельные ветки для каждой функциональности
3. Часто отправляйте изменения на сервер
4. Регулярно синхронизируйтесь с основной веткой
