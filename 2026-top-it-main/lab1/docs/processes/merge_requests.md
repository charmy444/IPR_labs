# Merge Requests

## Что такое Merge Request?

Merge Request (MR) — это запрос на слияние изменений из одной ветки в другую. В GitLab Merge Request — это основной механизм для код-ревью, обсуждения изменений и контроля качества кода.

## Создание Merge Request

### Через веб-интерфейс GitLab

1. После отправки изменений на сервер GitLab автоматически предложит создать Merge Request
2. Или перейдите в раздел "Merge Requests" и нажмите "New merge request"
3. Выберите:
   - **Source branch** — ваша feature-ветка
   - **Target branch** — ветка, в которую нужно слить изменения (обычно `main`)
4. Нажмите "Compare branches and continue"

### Заполнение информации о Merge Request

#### Заголовок

Создайте информативный заголовок, отражающий суть изменений:
- ✅ "Добавлена валидация формы регистрации"
- ❌ "Исправления" или "Update"

#### Описание

Подробно опишите:
1. **Что было сделано**: Краткое описание реализованной функциональности
2. **Зачем это нужно**: Проблема, которую решает этот MR
3. **Как проверить**: Пошаговая инструкция для тестирования изменений
4. **Скриншоты** (если применимо): Визуальные изменения

Пример описания:

```
## Что было сделано
Добавлена валидация email при регистрации пользователя

## Зачем это нужно
Ранее пользователи могли вводить некорректный email, что приводило к ошибкам при отправке уведомлений

## Как проверить
1. Перейдите на страницу регистрации
2. Введите некорректный email (например, "test")
3. Убедитесь, что появляется сообщение об ошибке
4. Введите корректный email
5. Убедитесь, что регистрация проходит успешно

## Связанные задачи
Closes #123
```

### Настройки Merge Request

#### Assignee (Назначенный)

Выберите человека, ответственного за ревью и слияние этого MR. Обычно это владелец репозитория или технический руководитель.

#### Reviewers (Ревьюеры)

Добавьте коллег, которые должны провести код-ревью. В учебных проектах это обычно ваш напарник.

#### Labels (Метки)

Используйте метки для классификации MR:
- `feature` — новая функциональность
- `bugfix` — исправление ошибок
- `documentation` — изменения в документации
- `refactor` — рефакторинг кода

#### Milestone (Этап)

Если проект использует milestones для планирования, укажите соответствующий этап.

## Процесс код-ревью

### Роли в код-ревью

#### Автор MR

Ответственности:
- Создание качественного MR с подробным описанием
- Ответы на комментарии ревьюеров
- Внесение требуемых изменений
- Поддержание диалога с ревьюерами

#### Ревьюер

Ответственности:
- Проверка кода на соответствие стандартам
- Поиск потенциальных ошибок и проблем
- Предложение улучшений
- Утверждение или отклонение MR

### Что проверять при код-ревью

#### Функциональность

- Корректность реализации
- Обработка граничных случаев
- Наличие и качество тестов

#### Кодстайл

- Следование принятому стилю кодирования
- Читаемость и понятность кода
- Наличие комментариев к сложным участкам

#### Архитектура

- Соответствие архитектурным принципам проекта
- Правильное использование паттернов проектирования
- Минимизация связанности компонентов

#### Безопасность

- Отсутствие уязвимостей
- Правильная обработка пользовательского ввода
- Безопасное хранение данных

### Комментарии в код-ревью

#### Как оставлять комментарии

1. Перейдите на вкладку "Changes"
2. Наведите курсор на строку кода
3. Нажмите на знак "+" для добавления комментария
4. Напишите конструктивный комментарий

#### Тон комментариев

- Будьте вежливы и уважительны
- Объясняйте, почему предложено изменение
- Предлагайте альтернативные решения
- Используйте конструктивный тон

Пример хорошего комментария:

```
Здесь лучше использовать метод `validateEmail()` из утилит, чтобы избежать дублирования логики валидации. Что думаешь?
```

Пример плохого комментария:

```
Это неправильно, переделай.
```

## Слияние Merge Request

### Требования для слияния

1. Все ревьюеры утвердили изменения
2. Все автоматические проверки прошли успешно
3. Нет неразрешенных конфликтов

### Опции слияния

#### Merge commit

Создает отдельный коммит слияния. История остается полной, но становится более сложной.

#### Fast-forward merge

Перемещает указатель ветки вперед. История остается линейной, но теряется информация о MR.

#### Rebase and merge

Перебазирует коммиты поверх целевой ветки. История остается линейной и чистой.

#### Squash and merge

Объединяет все коммиты в один. История остается чистой, но теряется детальная информация о разработке.

### Рекомендации по слиянию

- Используйте "Delete source branch" после слияния для поддержания чистоты репозитория
- Убедитесь, что все комментарии разрешены перед слиянием
- Проверьте, что описание MR содержит всю необходимую информацию

## Автоматизация и CI/CD

### Pipeline статус

Merge Request показывает статус CI/CD pipeline:
- ✅ Success — все проверки пройдены
- ❌ Failed — есть ошибки, требующие исправления
- ⏳ Running — проверки выполняются
- 🚫 Skipped — некоторые проверки пропущены

### Автоматические проверки

Типичные автоматические проверки:
- Запуск unit-тестов
- Проверка кодстайла (linting)
- Анализ безопасности
- Статический анализ кода

## Best Practices

### 1. Создавайте маленькие MR

Чем меньше MR, тем легче его ревьюировать. Один MR — одна логическая задача.

### 2. Пишите качественные описания

Хорошее описание экономит время ревьюеров и помогает в будущем понять, зачем были внесены изменения.

### 3. Отвечайте на комментарии

Не игнорируйте комментарии ревьюеров. Даже если вы не согласны, объясните свою точку зрения.

### 4. Используйте шаблоны MR

Создайте шаблон для MR, чтобы стандартизировать описание и не забывать важные пункты.
