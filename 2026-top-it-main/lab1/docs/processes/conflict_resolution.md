
# Разрешение конфликтов слияния

## Что такое конфликты слияния?

Конфликты слияния возникают, когда Git не может автоматически объединить изменения из разных веток, потому что одни и те же части файлов были изменены по-разному. Это нормальная ситуация в совместной разработке и важно уметь их разрешать.

## Причины конфликтов

### Одновременные изменения одной и той же строки

Когда два разработчика изменяют одну и ту же строку кода в разных ветках.

### Изменение и удаление одного файла

Когда один разработчик изменяет файл, а другой удаляет его.

### Разные форматирования кода

Когда разработчики используют разные настройки редактора (пробелы vs табы).

## Обнаружение конфликтов

### При слиянии веток

```bash
git merge feature/branch-name
```

Если возникает конфликт, Git выводит сообщение:

```
Auto-merging file-name.js
CONFLICT (content): Merge conflict in file-name.js
Automatic merge failed; fix conflicts and then commit the result.
```

### При перебазировании

```bash
git rebase main
```

При конфликте перебазирование останавливается:

```
First, rewinding head to replay your work on top of it...
Applying: commit message
Using index info to reconstruct a base tree...
M	file-name.js
Falling back to patching base and 3-way merge...
Auto-merging file-name.js
CONFLICT (content): Merge conflict in file-name.js
error: Failed to merge in the changes.
Patch failed at 0001 commit message
```

## Идентификация конфликтующих файлов

Просмотреть статус репозитория для выявления конфликтов:

```bash
git status
```

Конфликтующие файлы отмечаются как:

```
Unmerged paths:
  (use "git add/rm <file>..." as appropriate to mark resolution)
	both modified:      file-name.js
```

## Ручное разрешение конфликтов

### Просмотр конфликтов

Откройте конфликтующий файл в текстовом редакторе. Вы увидите специальные маркеры:

```javascript
<<<<<<< HEAD
console.log("Это изменение из основной ветки");
=======
console.log("Это изменение из feature-ветки");
>>>>>>> feature/new-feature
```

### Структура маркеров конфликта

- `<<<<<<< HEAD` — начало конфликта, содержимое из текущей ветки
- `=======` — разделитель между двумя версиями
- `>>>>>>> feature/new-feature` — конец конфликта, содержимое из ветки, которую мы сливаем

### Разрешение конфликта

1. **Выберите нужную версию**:
   - Оставьте одну из версий и удалите другую вместе с маркерами
   - Объедините изменения вручную

2. **Удалите маркеры конфликта**:
   - Удалите строки `<<<<<<<`, `=======`, и `>>>>>>>`

3. **Пример разрешения**:

Было:
```javascript
<<<<<<< HEAD
console.log("Это изменение из основной ветки");
=======
console.log("Это изменение из feature-ветки");
>>>>>>> feature/new-feature
```

Стало:
```javascript
console.log("Это объединенное изменение из обеих веток");
```

### Завершение разрешения

После ручного разрешения всех конфликтов:

1. Добавьте измененные файлы в индекс:

```bash
git add resolved-file.js
```

2. Завершите слияние:

```bash
git commit
```

Или при перебазировании:

```bash
git rebase --continue
```

## Использование инструментов для разрешения конфликтов

### Встроенные инструменты Git

Git предоставляет встроенные инструменты для разрешения конфликтов:

```bash
# Открыть конфликтующий файл в редакторе по умолчанию
git mergetool
```

### Графические инструменты

Популярные графические инструменты:
- **VS Code** — встроенный инструмент разрешения конфликтов
- **Sublime Merge** — специализированный Git-клиент
- **GitKraken** — графический Git-клиент с удобным интерфейсом

### Настройка инструмента разрешения конфликтов

Установите предпочитаемый инструмент:

```bash
# Для VS Code
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'

# Для Sublime Merge
git config --global merge.tool smerge
git config --global mergetool.smerge.cmd 'smerge mergetool "$BASE" "$LOCAL" "$REMOTE" -o "$MERGED"'
```

## Профилактика конфликтов

### Частая синхронизация

Регулярно обновляйте свою ветку относительно основной:

```bash
git fetch origin
git rebase origin/main
```

### Маленькие и частые коммиты

Чем меньше изменения в каждом коммите, тем меньше вероятность конфликтов.

### Коммуникация в команде

Согласовывайте работу над пересекающейся функциональностью.

## Отмена слияния или перебазирования

### Отмена слияния

Если слияние прошло неудачно:

```bash
git merge --abort
```

### Отмена перебазирования

Если перебазирование прошло неудачно:

```bash
git rebase --abort
```

## Продвинутые техники

### Использование git checkout для выбора сторон

При конфликте можно выбрать одну из сторон полностью:

```bash
# Выбрать нашу версию (из текущей ветки)
git checkout --ours file-name.js

# Выбрать их версию (из сливаемой ветки)
git checkout --theirs file-name.js

# Затем добавить файл в индекс
git add file-name.js
```

### Поиск конфликтов перед слиянием

Проверьте, будут ли конфликты при слиянии:

```bash
# Проверить, можно ли выполнить fast-forward merge
git merge --ff-only branch-name

# Проверить, будет ли конфликт (без фактического слияния)
git merge --no-commit --no-ff branch-name
# Если есть конфликты, отменяем
git merge --abort
```

## Best Practices

### 1. Решайте конфликты сразу

Не откладывайте разрешение конфликтов, так как они могут стать сложнее со временем.

### 2. Понимайте изменения

Перед разрешением конфликта убедитесь, что понимаете, что делают обе версии кода.

### 3. Тестируйте результат

После разрешения конфликтов обязательно протестируйте результат.

### 4. Коммуницируйте с командой

Если конфликт сложный, обсудите его с другими разработчиками.
